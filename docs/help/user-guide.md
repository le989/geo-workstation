# GEO 工作站内部用户快速上手

本文面向 GEO 工作站的内部使用者，帮助平台管理员、公司管理员和运营人员快速理解系统怎么用、数据怎么搭、日常怎么维护。本文不包含部署命令、数据库配置、真实账号密码或 API Key。

## 1. GEO 工作站是什么

GEO 工作站是一套面向生成式 AI 搜索和问答场景的营销运营系统。它不是普通内容管理系统，也不是单纯的知识库工具。

系统要解决的问题是：当用户向 AI 询问某类产品、服务、品牌、方案或选型问题时，品牌是否更容易被提及、推荐、引用，并且回答是否更准确可信。

日常工作围绕这条链路展开：

`公司与产品线 -> 部门和模块权限 -> 提示词策略库 -> AI 拓词 -> 知识库 -> 指令库 -> GEO 内容生成 -> 售后问答 -> AI 模型覆盖记录 -> GEO 报表 -> 使用统计和操作日志`

每个模块的作用：

- Dashboard：查看当前公司 GEO 资产、任务和待优化事项。
- GEO 诊断：前期诊断品牌、官网、产品线和关键词，发现提示词、知识库和内容缺口。
- 提示词策略库：维护用户可能向 AI 提问的问题和追踪词。
- AI 拓词：根据产品、服务和场景生成候选提示词，人工选择后入库。
- 部门管理：维护公司内部门、用户所属部门和模块进入权限。
- 知识库：维护产品、服务、案例、资质、FAQ、解决方案等事实资料，并标记资料类型、可靠程度和资料状态。
- 指令库：维护可复用的 GEO 内容生成指令模板。
- GEO 内容生成：基于提示词、知识库和指令模板创建内容任务并生成草稿。
- 售后问答：面向内部人员，基于已通过的售后资料和产品资料回答问题，并显示引用来源。
- AI 模型覆盖记录：人工录入或导入不同 AI 模型下的品牌提及、推荐和引用情况。
- GEO 报表：查看提示词、内容、知识库和模型覆盖表现，决定下一步补什么。
- 用户管理：由平台管理员维护全局账号；公司管理员可维护本公司普通用户、部门绑定和密码重置。
- 使用统计：由平台管理员和公司管理员查看 AI / mock / stub 调用统计。
- 操作日志：由平台管理员和公司管理员查看登录、部门、知识库、内容、拓词、报表和售后问答等关键操作。
- 系统设置：维护公司管理、产品线管理、项目档案和基础配置说明。
- 公司管理：新增、编辑、启用 / 停用公司。
- 产品线管理：新增、编辑、启用 / 停用产品线，可填写产品线说明补充用途、适用场景和内部识别信息。

## 2. 登录系统与角色说明

登录后，系统会根据账号角色和当前公司展示可访问的页面。请使用分配给自己的内部账号，不要共用账号。

当前正式角色：

- 平台管理员：对应 `platform_admin`。负责管理用户、公司、系统设置和全局能力，可以处理跨公司基础维护。
- 公司管理员：对应 `company_admin`。负责当前公司内的业务数据维护，包括本公司普通用户、部门绑定、提示词、知识库、指令库、内容任务和报表复盘。
- 运营人员：对应 `operator`。负责日常维护提示词、知识库、指令模板、内容任务、模型覆盖记录等运营工作。
- 查看者：对应 `viewer`。主要用于只读、演示、临时查看，不作为核心日常维护角色。

使用建议：

- 平台管理员先把公司、产品线、用户和权限维护好。
- 公司管理员负责确认当前公司的基础资料是否完整。
- 运营人员按日常 SOP 维护提示词、资料、内容和覆盖记录。
- 查看者只看数据和演示页面，不参与日常写入维护。
- 部门权限控制的是能不能进入模块；角色控制的是能不能写入。查看者即使进入某个模块，仍然保持只读。

## 3. 认识 Dashboard

入口：`/dashboard`

Dashboard 是日常进入系统后的第一站，用来快速判断当前公司 GEO 工作是否健康。

它主要用于查看：

- 当前公司提示词、知识库、内容和模型覆盖的整体情况。
- 哪些提示词缺内容。
- 哪些产品线缺知识库资料。
- 哪些内容任务失败或需要继续处理。
- 下一步应优先做诊断、补提示词、补资料、生成内容还是复盘报表。

下一步看哪里：

- 如果没有基础数据，先去系统设置确认公司和产品线。
- 如果缺少用户会问的问题，去提示词策略库和 AI 拓词。
- 如果缺少事实资料，去知识库。
- 如果已有提示词和知识库，去 GEO 内容生成。
- 如果已有发布或人工检测结果，去 AI 模型覆盖记录和 GEO 报表。

## 4. 第一次使用：确认公司和产品线

入口：`/settings`

正式使用前，先确认当前公司和产品线。clean 库业务数据保持 0 时，也应按这个顺序从 0 搭建正式数据。

推荐顺序：

1. 由平台管理员或公司管理员进入系统设置。
2. 在公司管理中新增或确认公司。
3. 编辑公司名称、编码等基础信息。
4. 根据实际情况启用或停用公司，不建议物理删除。
5. 在产品线管理中新增产品线。
6. 编辑产品线名称、编码、排序和状态。
7. 根据实际情况启用或停用产品线，不建议物理删除。
8. 再维护项目档案、品牌上下文、目标客户和表达边界。

注意：

- 公司管理支持新增、编辑、启用 / 停用。
- 产品线管理支持新增、编辑、启用 / 停用。
- 产品线说明用于补充这个产品线的用途、适用场景和内部识别信息，方便后续内容生成和知识引用。
- 公司和产品线是后续提示词、知识库、内容任务和报表复盘的基础，不建议跳过。
- 基础 seed 与演示 seed 已分离。clean 库从 0 搭建正式数据时，只保留基础账号、公司和产品线，不写入演示 GEO 业务数据。

## 5. 部门管理与模块访问权限

入口：`/departments`

部门管理用于控制公司内部不同部门能进入哪些模块。第一版一个用户在一个公司内只绑定一个部门。

平台管理员和公司管理员可以：

- 新增部门。
- 编辑部门名称和编码。
- 启用或停用部门。
- 配置部门可进入的模块。
- 在用户管理中给用户绑定部门。

权限关系：

- 部门模块权限只控制“能不能进入模块”。
- 角色仍然控制写入能力。
- 平台管理员和公司管理员不会因为部门权限配置错误被锁死。
- 运营人员和查看者受所属部门的模块权限控制。
- 查看者仍然只读，不能因为部门权限获得上传、编辑、删除等写入能力。

建议先完成部门和模块权限，再安排运营人员维护知识库、售后问答和内容生产。

## 6. 建立提示词策略库

入口：`/geo-prompts`

提示词策略库用于维护用户可能向 AI 提问的问题，也是后续内容生成和模型覆盖记录的基础。

什么时候用：

- 新公司或新产品线刚开始搭建时。
- 做完 GEO 诊断后，需要把建议问题沉淀下来时。
- 运营中发现新的用户问题、选型问题、对比问题时。

建议维护的提示词类型：

- 训练词：基础产品、服务、品牌、行业词。
- 蒸馏词：更像真实用户会问 AI 的完整问题。
- 品牌词：围绕品牌名称、品牌能力、品牌可信度的问题。
- 场景词：围绕具体应用场景、需求决策和行动前准备的问题。

好的提示词通常不是单个关键词，而是具体问题。例如：

- 某类产品适合哪些应用场景？
- 选择某类服务前需要确认哪些条件？
- 某品牌和替代方案相比有什么优势？
- 某类解决方案有哪些常见风险？

下一步看哪里：

- 如果需要扩展更多候选问题，去 AI 拓词。
- 如果提示词已经确定，去知识库补充事实资料。
- 如果提示词和知识库都准备好，去 GEO 内容生成。

## 7. 使用 AI 拓词生成候选提示词

入口：`/expansion`

AI 拓词用于根据产品、服务、课程、门店、个人品牌方向和目标客户场景，生成更像真实用户会向 AI 提问的候选提示词。

推荐流程：

1. 进入 AI 拓词。
2. 输入产品、服务、课程、门店或个人品牌方向。
3. 补充目标客户、应用场景和限制条件。
4. 生成候选提示词。
5. 人工检查候选词是否像真实用户问题。
6. 去掉泛词、纯价格词、广告词和过宽的问题。
7. 勾选高价值候选词保存到提示词策略库。
8. 回到提示词策略库设置优先级和追踪状态。

注意：

- 候选词不会自动入库，必须人工勾选保存。
- 高价值词应围绕需求决策、问题诊断、对比替代、信任验证和行动前准备。
- AI 拓词历史清理属于后续能力，当前重点是把有价值候选词沉淀到提示词策略库。

## 8. 建立知识库

入口：`/knowledge-bases`

知识库是 GEO 内容生成的事实底座。它决定生成内容可以引用哪些企业资料，也影响内容是否可信、清楚、可复盘。

什么时候用：

- 新公司、新产品线或新服务线开始搭建时。
- 内容生成前发现资料不足时。
- 产品资料、服务说明、案例、资质或 FAQ 有更新时。

建议按产品线或业务主题建立知识库，并优先导入真实资料：

- 产品能力。
- 应用场景。
- 技术优势。
- 客户案例。
- 资质认证。
- 选型参数。
- 常见问题。
- 服务说明。
- 解决方案。
- 竞品对比或国产替代说明。

资料治理字段：

- 资料类型：产品资料、售后资料、公司可信信息、内容引用资料、内部制度 / 流程资料、客户案例资料。
- 资料主题：公司新闻、活动资讯、资质证书、培训资料、行业动态、选型资料、故障排查、安装接线、应用案例、品牌介绍、产品参数、售后流程等。
- 资料状态：待审核、已通过、已停用。
- 可靠程度：高可靠、中可靠、低可靠。
- 可用场景：例如内部检索、GEO 内容生成、售后问答、GEO 分析。
- 售后资料可以限制可见部门，避免普通用户跨部门查看不该看的售后资料。

注意：

- 知识库是事实底座，不要把测试数据和真实数据混在一起。
- 具体参数、认证、案例、价格等必须有依据。
- 不确定资料不要写成确定参数。
- 知识库、文件和知识片段支持软删除。
- 售后问答只使用已通过的售后资料和已通过的产品资料。
- 本地存储路径不会展示给前端用户。

## 9. 上传资料 / 手动录入资料

入口：`/knowledge-bases`

进入某个知识库后，通过“资料入库向导”补充资料。默认流程只保留核心字段：

1. 手动录入：填写资料标题、资料类型、所属目录和正文内容。
2. 文件上传：选择文件、资料类型和所属目录；资料标题可后补，不填时默认使用文件名。
3. 需要细分主题、调整可用场景、资料状态、可靠程度、来源说明或售后可见部门时，展开“高级资料属性”。
4. 系统会按资料类型自动填默认用途、待审核状态和可靠程度，普通入库不必一次填完全部字段。

建议做法：

1. 先确认资料属于哪个公司和产品线。
2. 给资料起清楚的标题。
3. 选择资料类型，不要把公司新闻、资质证书、行业动态误建成资料类型；这些应放在资料主题中。
4. 如需要更细分类，再展开高级资料属性选择资料主题，让后续筛选和引用更清楚。
5. 按需确认可用场景、可靠程度和资料状态；待审核资料和低可靠资料不会被售后问答或 GEO 内容正式引用。
6. 如果是售后资料，按需要配置可见部门。
7. 上传或录入前先去掉无关内容。
8. 每条知识片段尽量只讲一个主题。
9. 上传或保存后检查解析状态和片段内容。
10. 需要时编辑片段，让标题和正文更清楚。

当前更适合上传的文件类型：

- `txt`
- `md`
- `csv`
- `xlsx`
- `xls`
- `docx`

PDF、OCR 和图片文字识别属于后续能力。当前不要把 PDF/OCR 写成已支持能力。

如果 Word 文件很大，或文件名类似“知识库资料收集文档 / 综合资料包 / 正式版”，建议先拆成资料卡片后再上传。一条资料尽量只解决一个主题，例如公司介绍、资质证书、产品资料、售后资料、客户案例、FAQ 或行业知识。这样更利于售后问答、GEO 内容生成和内部查询准确引用。

说明：

- 目录表示资料放在哪个知识库中，方便人工整理。
- 资料类型表示资料的大类，影响系统判断资料性质。
- 资料主题表示更细的内容主题，例如故障排查、安装接线、资质证书。
- 可用场景决定资料可以服务哪些功能。
- 资料状态决定资料能否作为正式引用依据。
- 可靠程度决定资料能否进入正式引用范围；低可靠资料仍可内部查看和维护，但不会被售后问答或 GEO 内容生成正式引用。
- 资料列表支持简洁 / 管理视图切换，并可按关键词、目录、资料状态筛选；资料类型、资料主题、可靠程度、可用场景、解析状态和文件类型放在高级筛选里。
- “可被 AI 引用”表示资料已通过且可靠程度为高 / 中；“暂不可引用”会提示未审核、低可靠或已停用等原因。

## 10. 建立指令模板

入口：`/instruction-templates`

指令库用于维护可复用的 GEO 内容生成方法。指令模板不是普通写作提示，而是指导系统如何围绕提示词、知识库和 GEO 目标生成内容。

一个好的指令模板应说明：

- 内容服务哪个 GEO 目标。
- 要回答哪类用户问题。
- 必须引用哪些知识库事实。
- 哪些表达不能写。
- 是否需要 FAQ、列表、对比、选型建议或判断逻辑。
- 内容适合发布到官网、公众号、B2B 页面还是销售资料。

什么时候用：

- 需要稳定生成同类内容时。
- 需要统一内容语气和结构时。
- 需要让内容更适合被 AI 摘取时。

下一步看哪里：

- 如果已经有提示词、知识库和指令模板，就可以进入 GEO 内容生成。

## 11. 创建 GEO 内容任务

正式入口：`/geo-content`

兼容入口：`/content-tasks`。该入口保留用于历史兼容，日常教程和演示请使用 `/geo-content`。

GEO 内容生成用于把提示词、知识库和指令模板串起来，生成面向 AI 搜索优化的内容草稿。

推荐流程：

1. 从提示词策略库选择高优先级提示词。
2. 确认对应知识库资料足够。
3. 选择合适的指令模板。
4. 创建 GEO 内容任务。
5. 等待任务完成或查看失败原因。
6. 进入内容项查看草稿。

内容任务可以用于：

- 补齐某个提示词下的品牌可见度。
- 支撑某类用户向 AI 提问的场景。
- 补齐知识库或内容缺口。
- 形成官网、公众号、B2B 页面或销售资料的内容草稿。

注意：

- 内容生成结果必须人工审核。
- 参数、价格、认证、案例、效果承诺必须谨慎。
- GEO 内容任务支持归档，归档后默认列表中隐藏，但历史记录仍保留。

## 12. 查看、编辑、导出内容草稿

入口：`/geo-content`

内容任务完成后，可以查看内容项。每个内容项通常对应一个提示词或一个内容生成目标。

建议检查：

- 标题是否清楚。
- 正文是否回答了用户问题。
- 是否引用了知识库事实。
- 是否出现未经证实的参数、认证、协议或效果承诺。
- 是否有 FAQ、列表、判断逻辑或选型建议。
- 是否符合公司品牌语气和禁止表达。

常用动作：

- 编辑内容草稿。
- 删除不合适的内容项。
- 导出 Markdown。
- 做内容质量检查。
- 生成发布优化版。
- 生成富文本发布稿。

富文本发布稿用于复制到外部发布平台，但系统当前不做自动发布。

## 13. 使用售后问答

入口：`/aftersales-qa`

售后问答面向公司内部人员，不是客户开放版，也不是万能聊天机器人。当前页面是对话式售后知识助手：左侧显示历史会话，右侧显示用户问题和系统回答，回答必须基于已审核资料并带引用来源。

会话能力：

- 可以新建售后会话。
- 第一条问题会自动生成会话标题。
- 可以手动重命名会话。
- 可以搜索会话标题。
- 默认显示进行中的会话，支持查看已归档会话和全部会话。
- 可以归档会话，不做物理删除；已归档会话可查看和恢复。
- 已归档会话默认只读，恢复后可继续提问。
- 管理员可切换“我的会话 / 全部会话”，普通用户只看自己的会话。
- 左侧列表支持加载更多，适合会话较多时分批查看。
- 连续追问时，系统只使用上一轮问题和当前问题合并检索，不做复杂长上下文记忆。
- 不做流式输出，提问后显示加载中，完成后一次性返回回答。

回答依据：

- 优先使用已通过的售后资料。
- 如果没有命中可靠售后资料，再使用已通过的产品资料兜底。
- 不使用待审核、已停用、无权访问的售后资料。
- 不使用公司可信信息、内容引用资料、内部制度 / 流程资料、客户案例资料作为售后答案依据。

引用来源会显示：

- 知识库。
- 文件。
- 片段。
- 资料类型。
- 片段摘要。

当知识库中没有可靠依据时，页面会提示：

“未找到可引用资料，建议补充资料或转人工确认。”

当问题过于模糊时，页面会提示补充产品型号、现场现象、输出方式或接线情况，不会硬编答案。

当问题是“怎么补充资料”“人工确认是什么意思”“售后问答怎么用”等系统使用说明时，页面会返回简短使用引导，不会生成虚假的引用来源。

回答纠错反馈：

- 每条系统回答下方可以点击“回答有误”。
- 反馈时需要选择错误类型，例如引用错了、答案不完整、答案错误、知识库缺资料、问题描述不清或其他。
- “正确答案 / 补充说明”为必填，用于告诉管理员应如何复核或补充资料。
- 提交后该回答会显示“已反馈”状态，管理员可在售后问答页顶部切换到“反馈待处理”查看。
- 管理员可以标记“已处理”或“无需处理”，处理结果会回写到对应问答记录。
- 第一版反馈只用于复盘和知识库优化线索，不会自动写入知识库，也不会生成待审核资料草稿。

记录查看规则：

- 运营人员和查看者只能查看自己的会话和问答记录。
- 平台管理员和公司管理员可以查看本公司全部会话和问答记录。
- 运营人员和查看者只能提交自己的回答反馈；反馈待处理列表仅管理员可见。
- 售后问答会计入使用统计和操作日志。

## 14. 录入 / 导入 AI 模型覆盖记录

入口：`/model-inclusion-records`

AI 模型覆盖记录用于人工记录不同 AI 模型、入口或检测方式下，品牌是否被提及、推荐、引用官网，以及相对竞品的表现。

什么时候用：

- 发布内容后一段时间，人工检查 AI 回答表现时。
- 做阶段复盘，需要记录模型覆盖情况时。
- 从外部表格整理模型覆盖数据后，批量导入时。

建议记录：

- 关联提示词。
- AI 模型或平台。
- 品牌是否被提及。
- 品牌是否被推荐。
- 是否引用官网或可信资料。
- 推荐位置或备注。
- 检测时间。

注意：

- AI 模型覆盖记录当前暂不支持编辑 / 作废。录入前请先确认数据准确。
- 外部 AI 自动检测属于后续能力，当前以人工录入和导入为主。

## 15. 查看 GEO 报表

正式入口：`/geo-reports`

兼容入口：`/reports`。该入口保留用于历史兼容，日常教程和演示请使用 `/geo-reports`。

GEO 报表用于复盘当前公司 GEO 工作的整体情况。

重点看：

- 提示词数量和类型分布。
- 高优先级提示词是否已有内容支撑。
- 模型覆盖记录是否足够。
- 品牌提及率、推荐率和官网引用情况。
- 哪些产品线缺知识库资料。
- 哪些内容任务失败或需要补齐。
- 下一步优化建议。

下一步看哪里：

- 提示词不足：回到提示词策略库或 AI 拓词。
- 知识库不足：回到知识库。
- 内容不足：回到 GEO 内容生成。
- 模型覆盖不足：回到 AI 模型覆盖记录。

## 16. 查看使用统计和操作日志

使用统计入口：`/usage-analytics`

## 18. 日常运营建议

建议公司管理员和运营人员按周或按项目节奏维护：

1. 查看 Dashboard，确认当前公司是否有明显缺口。
2. 先建部门，配置部门模块访问权限，再给用户绑定部门。
3. 维护产品线下的高优先级提示词。
4. 用 AI 拓词补充候选提示词，人工筛选后入库。
5. 建知识库，通过资料入库向导上传或手动录入资料。
6. 设置资料类型；需要细分时在高级资料属性中补充资料主题、可靠程度、可用场景和资料状态。
7. 售后资料按需配置可见部门，审核通过后再用于售后问答。
8. 使用售后问答时查看引用来源，无依据时补充资料或转人工确认。
9. 维护指令模板，让内容结构更稳定。
10. 在 GEO 内容生成中创建少量高价值内容任务。
11. 审核、编辑和导出内容草稿。
12. 发布后人工记录 AI 模型覆盖情况。
13. 在 GEO 报表复盘提示词、内容、知识库和模型覆盖。
14. 通过使用统计和操作日志复盘使用情况。
15. 归档已完成的 GEO 诊断任务和 GEO 内容任务。

不建议一次性导入大量资料或批量生成大量内容。先跑通一个产品线，再复制到其他产品线。

## 19. 当前限制与注意事项

当前边界：

- AI 模型覆盖记录暂不支持编辑 / 作废。
- AI 拓词历史清理后置，当前重点是保存高价值候选词。
- 产品线说明字段用于补充用途、适用场景和内部识别信息，不替代正式知识库资料。
- company_admin 可管理本公司 operator / viewer、部门绑定和密码重置；不能管理平台管理员、其他公司用户或提权为公司 / 平台管理员。
- 售后问答反馈当前只做记录和管理员处理，不自动写入知识库；客户开放反馈中心后置。
- 转知识库待审核资料草稿属于后续 AQA-KB-LOOP-1，当前反馈不会自动生成草稿。
- 售后问答当前只面向内部人员，不做客户开放版。
- PDF / OCR 后置。
- Word 自动按标题拆分章节后置。
- 自动推荐资料类型 / 资料主题后置。
- token 额度限制和成本扣费后置。
- 查看者主要用于只读 / 演示，不是核心日常维护角色。
- 真实 AI Provider 需要单独配置，普通用户教程不包含 API Key 或部署配置。
- 系统当前不做自动发布。
- 外部 AI 自动检测属于后续能力。
- 发布链接和发布效果如需更细跟踪，可先用外部表格记录。

clean 库使用注意：

- clean 库用于从 0 搭建正式数据。
- clean 库业务数据应保持可控，不要导入演示 GEO 数据。
- 不要在 clean 库上运行会写入的 API 测试。
- 基础 seed 与演示 seed 已分离：基础 seed 只用于初始化必要账号、公司和产品线；演示 seed 只用于演示库，不用于正式 clean 库。

安全注意：

- 不要在普通教程、截图或交接文档里写真实账号密码。
- 不要写真实 API Key、token、Cookie。
- 不要写完整数据库连接串。
- 不要写真实服务器地址或真实内网 IP。
