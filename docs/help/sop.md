# GEO 工作站日常使用 SOP

本文用于规范 GEO 工作站的日常使用流程。建议平台管理员先完成基础管理和部门权限，公司管理员确认公司、产品线、部门和知识库治理规则，运营人员再维护提示词、知识库、售后问答、内容任务和模型覆盖记录。

## 1. 从 0 搭建正式数据的推荐顺序

适用场景：clean 库业务数据为 0，准备搭建正式公司数据。

推荐顺序：

`确认公司 -> 维护产品线 -> 建部门 -> 配置模块权限 -> 绑定用户部门 -> 建提示词 -> AI 拓词 -> 建知识库 -> 上传或手动录入资料 -> 审核资料 -> 建指令模板 -> 创建内容任务 -> 使用售后问答 -> 录入模型覆盖记录 -> 查看报表 -> 复盘统计和日志`

旧版最小链路可概括为：`确认公司 -> 维护产品线 -> 建提示词 -> AI 拓词 -> 建知识库 -> 建指令模板 -> 创建内容任务 -> 录入模型覆盖记录 -> 查看报表`。当前正式流程在此基础上增加部门权限、资料审核、售后问答、使用统计和操作日志。

详细步骤：

1. 平台管理员登录系统。
2. 进入系统设置。
3. 在公司管理中确认或新增公司。
4. 编辑公司基础信息，并确认公司为启用状态。
5. 在产品线管理中新增正式产品线。
6. 编辑产品线名称、编码、排序和状态。
7. 进入部门管理，新增正式部门。
8. 配置部门可进入的模块。
9. 在用户管理中给用户绑定部门；第一版一个用户在一个公司内只绑定一个部门。
10. 维护项目档案，补充品牌、官网、目标客户、内容语气和禁止表达。
11. 进入提示词策略库，新增或导入第一批正式提示词。
12. 进入 AI 拓词，为重点产品线生成候选提示词，人工勾选保存。
13. 进入知识库，建立第一个正式知识库。
14. 通过资料入库向导上传文件或手动录入 5-10 条真实核心资料。
15. 设置资料类型；需要细分或调整引用范围时展开高级资料属性，补充资料主题、可信度、适用模块和审核状态；售后资料按需配置可见部门。
16. 审核资料，通过后再用于售后问答或内容引用。
17. 进入指令库，创建或选择通用指令模板。
18. 进入 GEO 内容生成，创建第一批少量内容任务。
19. 如涉及内部售后问题，进入售后问答查看回答和引用来源。
20. 审核内容草稿，必要时编辑、导出或生成发布稿。
21. 进入 AI 模型覆盖记录，人工录入或导入第一批覆盖记录。
22. 进入 GEO 报表，查看提示词、知识库、内容和模型覆盖情况。
23. 由管理员查看使用统计和操作日志，复盘 AI 调用和关键操作。

注意：

- 基础 seed 与演示 seed 已分离。clean 库从 0 搭建时，不写入演示 GEO 业务数据。
- 不要在 clean 库上运行会写入的 API 测试。
- 公司 / 产品线是业务数据的基础，不建议跳过。
- 部门权限控制能否进入模块，角色仍控制写入权限；查看者仍然只读。
- 产品线当前暂不支持说明字段，详细业务说明可先写入项目档案或知识库。

## 2. 日常内容生产流程

适用角色：公司管理员、运营人员。

目标：围绕高价值提示词稳定生产可审核、可发布、可复盘的 GEO 内容。

步骤：

1. 进入 Dashboard，查看当前待优化事项。
2. 进入提示词策略库，选择高优先级提示词。
3. 确认该提示词所属公司和产品线。
4. 进入知识库，确认事实资料是否足够。
5. 进入指令库，选择适合的指令模板。
6. 进入 GEO 内容生成 `/geo-content` 创建内容任务。
7. 查看内容任务状态。
8. 打开内容项，审核标题、正文、FAQ、列表和判断逻辑。
9. 做内容质量检查。
10. 生成发布优化版或富文本发布稿。
11. 人工复制到外部发布平台。
12. 发布后记录链接和后续表现。
13. 进入 AI 模型覆盖记录，录入人工检测结果。
14. 进入 GEO 报表复盘。

注意：

- `/geo-content` 是 GEO 内容生成正式入口。
- `/content-tasks` 仅作为兼容入口。
- 不建议一次生成太多篇。
- 每篇发布前必须人工审核。
- 参数、认证、价格、效果承诺必须有依据。

## 3. GEO 诊断到内容补齐流程

适用角色：公司管理员、运营人员。

目标：从诊断发现缺口，再补提示词、补资料、补内容。

步骤：

1. 进入 GEO 诊断 `/geo-analysis`。
2. 创建诊断任务，填写品牌、官网、产品线和关键词。
3. 查看诊断结果中的提示词建议、知识库缺口和内容缺口。
4. 将可用提示词建议转入提示词策略库。
5. 对缺资料的产品线，进入知识库补充资料。
6. 对缺内容的提示词，进入 GEO 内容生成创建内容任务。
7. 内容审核后发布或导出。
8. 后续进入 AI 模型覆盖记录录入检测结果。
9. 进入 GEO 报表复盘补齐效果。
10. 诊断任务完成阶段用途后，可归档。

注意：

- GEO 诊断任务可归档。
- 归档不是物理删除，历史结果仍保留。
- 进行中的诊断任务不要随意归档。

## 4. 提示词维护流程

适用角色：公司管理员、运营人员。

目标：让提示词策略库始终反映真实用户会向 AI 提问的问题。

步骤：

1. 进入提示词策略库 `/geo-prompts`。
2. 按产品线、提示词类型、优先级和追踪状态筛选。
3. 新增高价值提示词。
4. 批量导入前先检查重复和格式。
5. 对无效、过期或明显不再维护的提示词执行软删除。
6. 进入 AI 拓词 `/expansion` 生成候选提示词。
7. 人工筛选候选词，保存到提示词策略库。
8. 为高价值词设置优先级和追踪状态。

注意：

- 提示词可软删除。
- 候选词不会自动入库。
- 优先保存具体问题，而不是普通关键词。
- 高价值词应围绕需求决策、问题诊断、对比替代、信任验证和行动前准备。

## 5. 部门与模块权限维护流程

适用角色：平台管理员、公司管理员。

目标：让不同部门只看到自己需要的模块，同时避免扩大写入权限。

步骤：

1. 进入部门管理 `/departments`。
2. 新增部门，填写清楚的部门名称和编码。
3. 按实际业务需要编辑或停用部门。
4. 为部门配置可进入的模块，例如知识库、售后问答、GEO 内容生成等。
5. 进入用户管理，为用户绑定所属部门。
6. 运营人员和查看者重新登录或刷新后，按部门模块权限看到对应菜单。
7. 平台管理员和公司管理员定期复核部门权限，避免过度开放。

注意：

- 第一版一个用户在一个公司内只绑定一个部门。
- 部门权限只控制进入模块，不控制按钮级或字段级权限。
- 平台管理员和公司管理员不会被部门权限锁死。
- 查看者仍然只读，不能因为部门权限获得上传、编辑或删除能力。

## 6. 知识库维护流程

适用角色：公司管理员、运营人员。

目标：维护可靠事实资料，减少内容生成和售后问答时的无依据表达。

步骤：

1. 进入知识库 `/knowledge-bases`。
2. 按公司、产品线或业务主题建立知识库。
3. 优先导入真实资料。
4. 进入“新增资料”，选择手动录入或文件上传；普通入库只需填写核心字段。
5. 上传 `txt`、`md`、`csv`、`xlsx`、`xls`、`docx`，或手动录入整理后的文本。
6. 设置资料类型：产品资料、售后资料、公司可信信息、内容引用资料、内部制度 / 流程资料、客户案例资料。
7. 需要细分资料内容时展开高级资料属性，设置资料主题：公司新闻、活动资讯、资质证书、培训资料、行业动态、选型资料、故障排查、安装接线、应用案例、品牌介绍、产品参数、售后流程等。
8. 按需调整可信度、适用模块和审核状态；系统会根据资料类型自动给出默认值。
9. 售后资料按需配置可见部门。
10. 检查文件解析状态。
11. 检查知识片段标题、正文、资料类型和产品线。
12. 需要时编辑片段内容。
13. 删除或软删除错误、过期、测试性质的资料。
14. 内容生成或售后问答前确认相关资料足够，并优先使用已通过资料。

注意：

- 知识库 / 文件 / 片段可软删除。
- 知识库是事实底座。
- 目录表示资料放在哪个知识库；资料类型表示大类；资料主题表示更细的内容主题；适用模块决定资料服务哪些功能。
- 不要把测试数据和真实数据混在一起。
- 不确定参数不要写成确定结论。
- 售后问答只使用已通过的售后资料和已通过的产品资料。
- PDF / OCR 后置，不要把它们写成当前已支持格式。
- 本地存储路径不对前端展示。

## 7. 指令模板维护流程

适用角色：公司管理员、运营人员。

目标：沉淀可复用的 GEO 内容生成方法，让内容结构和质量更稳定。

步骤：

1. 进入指令库 `/instruction-templates`。
2. 查看已有指令模板是否满足当前内容目标。
3. 新增或复制指令模板。
4. 明确模板适用的内容类型、提示词类型和目标模型。
5. 在模板中写清楚事实引用、结构要求和禁止表达。
6. 内容生成前选择合适模板。
7. 如果生成内容多次出现同类问题，回到指令库调整模板。

注意：

- 指令模板可软删除。
- 指令模板是方法，不是事实来源。
- 事实仍以知识库、提示词和任务输入为准。

## 8. 售后问答使用流程

适用角色：公司管理员、运营人员、查看者。

目标：让内部售后问题在对话式售后知识助手中基于已审核资料回答，有依据就给引用，无依据就提示补资料或转人工确认。

步骤：

1. 进入知识库，确认售后资料或产品资料已录入。
2. 将资料审核状态设为已通过；待审核和已停用资料不能作为售后问答依据。
3. 对售后资料配置可见部门。
4. 进入售后问答 `/aftersales-qa`。
5. 新建会话，或从左侧历史会话中继续追问。
6. 会话较多时，可搜索会话标题，或切换进行中、已归档、全部会话。
7. 管理员可在“我的会话 / 全部会话”之间切换；运营人员和查看者只查看自己的会话。
8. 不再需要的会话可以归档；归档后默认列表不显示，可在已归档中查看和恢复。
9. 已归档会话默认只读，恢复后再继续提问。
10. 输入内部售后问题，等待系统完成检索后一次性返回回答。
11. 查看回答内容和回答下方的引用来源折叠区。
12. 如果有引用来源，核对知识库、文件、片段和资料类型是否正确。
13. 连续追问时，系统只使用上一轮问题和当前问题合并检索；不要把它当作复杂长上下文聊天机器人。
14. 如果提示“未找到可引用资料，建议补充资料或转人工确认。”，不要把它当作确定答案。
15. 如果提示需要补充产品型号、现场现象、输出方式或接线情况，应先补充信息再继续。
16. 如果询问怎么补充资料或怎么转人工，系统会给出简短使用引导，不会生成引用来源。
17. 需要补资料时，回到知识库补充并审核后再提问。

注意：

- 售后问答是内部对话助手，不是客户开放版。
- 售后资料优先，产品资料兜底。
- 必须只使用已通过资料。
- 不能当万能聊天机器人使用。

## 9. AI 模型覆盖记录维护流程

适用角色：公司管理员、运营人员。

目标：记录不同 AI 模型或入口下品牌是否被提及、推荐和引用，为报表复盘提供依据。

步骤：

1. 进入 AI 模型覆盖记录 `/model-inclusion-records`。
2. 选择关联提示词。
3. 填写模型、入口、检测时间和检测结果。
4. 记录品牌是否被提及。
5. 记录品牌是否被推荐。
6. 记录是否引用官网或可信资料。
7. 如有外部表格，可批量导入。
8. 导入后进入 GEO 报表查看覆盖变化。

注意：

- AI 模型覆盖记录暂不支持编辑 / 作废。
- 录入前请先确认数据准确。
- 外部 AI 自动检测属于后续能力。

## 10. GEO 报表复盘流程

适用角色：平台管理员、公司管理员、运营人员、查看者。

目标：用报表判断下一步补词、补资料、补内容还是补覆盖记录。

步骤：

1. 进入 GEO 报表 `/geo-reports`。
2. 查看总览指标。
3. 查看提示词覆盖情况。
4. 查看模型覆盖情况。
5. 查看内容覆盖情况。
6. 查看知识库覆盖情况。
7. 查看优化建议。
8. 根据缺口回到对应模块处理。

判断方式：

- 提示词少：去提示词策略库和 AI 拓词。
- 知识库弱：去知识库补资料。
- 内容少：去 GEO 内容生成创建任务。
- 覆盖记录少：去 AI 模型覆盖记录录入。
- 旧任务太多：归档 GEO 诊断任务或 GEO 内容任务。

注意：

- `/geo-reports` 是 GEO 报表正式入口。
- `/reports` 仅作为兼容入口。

## 11. 使用统计与操作日志复盘流程

适用角色：平台管理员、公司管理员。

目标：复盘 AI / mock / stub 调用情况，并用操作日志排查关键操作。

步骤：

1. 进入使用统计 `/usage-analytics`。
2. 查看总调用、总 token、mock 调用、真实调用、成功和失败数量。
3. 按用户、部门、模块和时间趋势查看使用情况。
4. 进入操作日志 `/operation-logs`。
5. 按模块、动作、用户、部门、时间和结果筛选日志。
6. 重点查看登录、部门、知识库、内容生成、AI 拓词、模型覆盖、报表导出、售后问答等关键动作。
7. 如发现异常，回到对应模块核对数据。

注意：

- mock / stub 调用 token 记为 0。
- 第一版不做额度限制，不做成本扣费。
- 第一版仅平台管理员和公司管理员查看全局统计和操作日志。
- 操作日志不记录密码、JWT、API Key、数据库连接串、本地路径、完整 prompt 或完整 AI 原文。

## 12. 归档与清理规则

建议规则：

- GEO 诊断任务：可归档。适合清理已完成、已复盘或不再关注的诊断任务。
- GEO 内容任务：可归档。适合清理已完成、已处理或不再继续执行的内容任务。
- 提示词：可软删除。适合处理重复、无效、过期或误导性提示词。
- 知识库 / 文件 / 片段：可软删除。适合处理错误、过期、测试或不应再引用的资料。
- 指令模板：可软删除。适合处理过期或不再使用的模板。
- 用户：建议禁用，不建议物理删除。
- 公司 / 产品线：启用 / 停用，不建议物理删除。

归档和软删除的目的，是让默认列表更清楚，同时保留必要历史痕迹。

## 13. 不建议操作

不建议：

- 在 clean 库上运行会写入的 API 测试。
- 把演示 seed 写入正式 clean 库。
- 一次性导入大量未经整理的资料。
- 把测试数据和真实数据混在同一个知识库。
- 直接发布未经质量检查的生成内容。
- 在普通教程或交接文档中写真实账号、密码、API Key、token、Cookie、完整数据库连接串、真实服务器地址或真实内网 IP。
- 给查看者安排日常维护任务。
- 把售后问答当作客户开放入口或万能聊天机器人。
- 在知识库中把待审核资料直接用于售后回答。
- 在操作日志中粘贴密码、JWT、API Key、数据库连接串、本地路径、完整 prompt 或完整 AI 原文。
- 把 `/content-tasks` 和 `/reports` 当成主入口对外演示。
- 在产品线说明字段尚未支持前，强行把说明写入无关字段。
- 把项目档案当成事实来源。项目档案只提供品牌和语气上下文，事实仍以知识库为准。
- 把真实 AI Provider、客户开放版、反馈中心、自动纠错入库、PDF/OCR、token 额度限制写成当前已完成能力。
